# 화면 정의서 & Checkpoint 기반 화면 흐름 정리

화면 정의서는 각 화면의 UI를 나열하는 문서가 아니라  
사용자의 상태와 전제 조건에 따라 화면 접근과 기능 흐름을 설명하는 문서이다.

본 문서는 화면 정의서를 작성하며 정리한  
화면 분류 기준, UI 넘버링 기준, Checkpoint 개념을 중심으로 정리한다.

---

## 1) 화면 정의서의 역할

화면 정의서는 다음 내용을 명확히 하기 위한 문서이다.

- 어떤 화면이 존재하는지
- 어떤 조건에서 접근 가능한지
- 화면 간 이동 흐름은 어떻게 구성되는지
- 특정 상태에서 기능이 제한되거나 활성화되는 이유는 무엇인지

즉, UI 자체보다 **사용자 흐름과 기능 접근 조건을 설명하는 데 목적이 있다.**

---

## 2) 화면 번호(UI 넘버링) 기준

화면 정의서에서의 UI 번호는  
UI 요소의 개수를 의미하지 않는다.

UI 번호는 **설명이 필요한 상호작용 포인트**를 식별하기 위한 용도로 사용된다.

### 번호를 부여하는 기준
- 버튼 클릭 시 화면 이동이 발생하는 경우
- 사용자 상태에 따라 분기되는 지점
- 기능 활성/비활성 조건이 존재하는 요소

### 번호를 생략해도 되는 경우
- 단순 입력 필드
- 정적인 텍스트, 라벨 요소

UI 번호는 화면 흐름 설명을 보조하는 수단이며,  
UI 설계서처럼 모든 요소를 세분화할 필요는 없다.

---

## 3) 화면 대분류 기준 설정

화면 분류는 탭 구조가 아니라  
**사용자 맥락과 기능 목적**을 기준으로 설정한다.

### 최종 대분류 구조
- **AUTH**: 로그인 / 회원가입
- **HOME**: 메인 대시보드 및 상태 요약
- **CARE**: 아이 케어 기능
- **SET**: 설정 및 정보 관리

---

### CARE로 리포트와 보이스를 묶은 이유

- 두 기능 모두 아이와의 상호작용 및 기록이라는 동일한 목적을 가짐
- 탭은 분리되어 있으나 기능적 맥락은 동일
- 화면 정의서 구조가 단순해지고 이해가 쉬워짐

이에 따라 리포트 탭과 보이스 탭은  
CARE 대분류 하위 화면으로 구성한다.

---

## 4) Checkpoint 개념 정의

화면 접근 조건을 명확히 하기 위해  
**Checkpoint(전제 조건)** 개념을 사용한다.

Checkpoint는 사용자의 상태를 기준으로  
접근 가능한 화면과 기능을 구분하는 기준이다.

### 서비스의 주요 Checkpoint
1. 로그인 여부
2. 기기 등록 여부
3. 아이 프로필 등록 여부
4. 리포트 생성 여부

위 네 가지 조건을 기준으로  
전체 화면 흐름을 설명할 수 있다.

---

## 5) Checkpoint 기반 화면 분기 구조

### 로그인 여부
- 미로그인 상태: 로그인 및 회원가입 화면만 접근 가능
- 로그인 완료 상태: HOME 화면 접근 가능

### 기기 등록 여부
- 기기 미등록: 기기 등록 화면으로 유도
- 기기 등록 완료: 아이 프로필 등록 가능

### 아이 프로필 등록 여부
- 아이 미등록: CARE 기능 접근 제한
- 아이 등록 완료: 리포트 및 보이스 기능 접근 가능

### 리포트 생성 여부
- 미생성: 요약 리포트 생성 버튼 노출
- 생성 완료: 타임라인 및 보이스 기능 활성화

Checkpoint는 화면 정의서 후반부에서  
QA, 개발, 기획 단계의 기준 역할을 수행한다.

---

## 6) 화면 정의서 구성 시 고려사항

- 화면 수보다 화면 간 흐름이 더 중요
- 상태 기반 분기를 문서로 설명할 수 있어야 함
- 화면 정의서는 구현 이전의 기준 문서 역할을 함

정리된 화면 정의서는  
기획 의도를 공유하고 구현 단계에서의 혼선을 줄이는 데 목적이 있다.

---

## 요약

화면 정의서는 UI 설명 문서가 아니라  
사용자 상태에 따른 화면 흐름과 기능 접근을 정의하는 문서이다.
